# 付録D. AIコーディングエージェント頻出用語・フレーズ集

AIコーディングエージェント（Claude Code、Codex等）は、Pythonのコーディング中にプログラミング用語やカタカナ語を説明なしに使う。プログラミング初心者がエージェントの出力で見慣れない語に遭遇したとき、本付録で素早く意味を引けるようにした。バイオインフォマティクス固有の用語は含めていない。これらはAIエージェントが利用するモデルやそのバージョン、Reasoning モードによって異なる点に注意が必要である。

本書の本文に登場する用語の定義は[用語集](./glossary.md)を参照。

---

## D-1. エージェントの定番フレーズ「翻訳」辞典

エージェントの出力は丁寧で自信ありげに見えるが、額面どおりに受け取ると痛い目に遭う表現がある。以下の「翻訳」を知っておくと、エージェントの出力を冷静に読めるようになる。

### 提案・推奨

| エージェントの言い方 | 実際の意味 |
|---|---|
| 「ベストプラクティスです」 | 一つの有力なやり方（唯一の正解ではない） |
| 「一般的には〜」 | よくある方法だがプロジェクトに合うかは別 |
| 「推奨されません」 | 使うと問題が起きる可能性があるが、必要な場面もある |
| 「エレガントな解決策です」 | 短くてきれいだが、読みにくい可能性もある |
| 「よりモダンなアプローチです」 | 新しいだけで必ずしも良いわけではない |
| 「〜はアンチパターンです」 | やるべきでないという主張だが、文脈次第では許容される |
| 「〜がデファクトスタンダードです」 | 広く使われているが、唯一の選択肢ではない |
| 「シンプルに保ちましょう」 | 複雑な解決策を思いつかなかった可能性もある |

### 自信度の読み方

| エージェントの言い方 | 実際の意味 |
|---|---|
| 「これで動くはずです」 | テストしていない |
| 「些細な変更です」 | 影響範囲は自分で確認すべき |
| 「パフォーマンスが向上します」 | 測定はしていない |
| 「環境によって異なります」 | 自分の回答に自信がない |
| 「詳しくはドキュメントを参照してください」 | 正確な答えを知らない |
| 「〜の可能性があります」 | 確信がない |
| 「おそらく〜です」 | 推測である |

### コード提示の前後

| エージェントの言い方 | 補足 |
|---|---|
| 「以下のように実装できます」 | コード提示の最頻出導入。動作確認はしていない |
| 「以下が修正版です」 | 差分を注意深く確認すべき |
| 「最小限の実装例です」 | 省略された部分がある |
| 「ステップバイステップで見ていきましょう」 | 長い説明が始まる合図。本当に全部読む必要があるか判断する |

### Claude特有の傾向

| 特徴 | 例 |
|---|---|
| 率直に限界を認める | 「正直に言うと、〜には自信がありません」 |
| 長文化しやすい | 聞いていない背景知識まで説明する |
| 構造化が好き | すぐ見出し・箇条書き・表にする |
| 安全側に倒す | 「〜のリスクがあるため、〜を推奨します」 |
| 代替案を必ず出す | 「〜もありますが、〜という選択肢もあります」 |

### ChatGPT/Codex特有の傾向

| 特徴 | 例 |
|---|---|
| 自信ありげに断言 | 不確かでも「〜です」と言い切る |
| 楽観的に締める | 「これで問題なく動作するはずです！」 |
| 同じ提案を繰り返す | 修正が効かないとき同じコードを再提示する |
| 「いい質問ですね！」 | 質問の良し悪しに関係なく使う |

## D-2. カタカナ動詞

エージェントは英語の技術用語をそのままカタカナにして「〜する」を付けて使う。以下はPythonのコーディング中に特に頻出するもの。

| 用語 | 意味 | エージェントの使い方の例 |
|---|---|---|
| パースする | 構文解析する | 「JSONをパースして必要なフィールドを取り出します」 |
| バリデーションする | 入力値の妥当性を検証する | 「ユーザー入力をバリデーションしてください」 |
| シリアライズする | データを保存・送信可能な形式に変換する | 「オブジェクトをJSONにシリアライズします」 |
| フェッチする | データを取得する | 「APIからデータをフェッチします」 |
| ラップする | 別の処理で包む | 「try-catchでラップしてエラーを捕捉します」 |
| ハンドリングする | 処理・対処する | 「エラーハンドリングを追加しましょう」 |
| モックする | テスト用に偽の実装を用意する | 「外部APIをモックしてテストします」 |
| スタブする | テスト用の仮実装を差し込む | 「この関数をスタブして固定値を返します」 |
| アサートする | テストで期待値と比較する | 「戻り値が正しいかアサートします」 |
| キャッシュする | 結果を一時保存して再利用する | 「レスポンスをキャッシュして高速化します」 |
| エンコードする | データを特定の形式に変換する | 「文字列をUTF-8でエンコードします」 |
| ノーマライズする | 値の範囲を揃える | 「特徴量をノーマライズします」 |
| フィルタリングする | 条件で絞り込む | 「低品質なデータをフィルタリングします」 |
| サンプリングする | データを抽出する | 「ランダムにサンプリングします」 |
| リファクタリングする | コードの振る舞いを変えず構造を改善する | 「この関数をリファクタリングしましょう」 |
| デプロイする | 本番環境に配置・公開する | 「変更をデプロイする前にテストしてください」 |
| スキャフォールドする | プロジェクトの初期構造（ディレクトリ、設定ファイル等）を自動生成する | 「プロジェクトをスキャフォールドします」 |

## D-3. カタカナ名詞

エージェントが説明なしに使う名詞。Pythonのコーディングで遭遇頻度が高いものを選んだ。

### コード構造・設計

| 用語 | 意味 |
|---|---|
| ボイラープレート | お決まりの定型コード。毎回書く必要があるが中身はほぼ同じ |
| デコレータ（デコレーター） | 関数やクラスに機能を付加するPythonの構文（`@` で始まる） |
| イテレーター | 要素を1つずつ順に取り出す仕組み |
| ジェネレーター | 値を逐次生成する仕組み。メモリ効率が良い |
| フィクスチャ | pytest がテスト前に用意する共通データや共通オブジェクト |
| コールバック | 後から呼び出される関数 |
| ハンドラー | イベントやリクエストを受け取って処理する関数 |
| ペイロード | 送受信するデータの本体部分 |
| エンドポイント | APIの接続先URL |
| スキーマ | データの構造定義 |
| コンテキスト | 実行環境や状態の受け渡し。Pythonの `with` 文で使うコンテキストマネージャーも関連 |
| ステータスメッセージ | 今どの処理段階にいるかを短く伝える進捗用メッセージ |
| ユーティリティ関数 | 汎用的な補助関数 |
| ヘルパー関数 | 特定の処理を助ける小さな関数 |
| ガード節 | 異常系を関数の先頭で弾いて早期リターンする書き方（guard clause） |
| エントリーポイント | プログラムの実行開始点。`if __name__ == "__main__"` やCLIの起動スクリプト |

### 状態・性質

| 用語 | 意味 |
|---|---|
| イミュータブル / ミュータブル | 変更不可能 / 変更可能。Pythonではタプルはイミュータブル、リストはミュータブル |
| イディオマティック | その言語らしい書き方。「Pythonic」はPythonにおけるイディオマティックの意 |
| スコープ | 変数が有効な範囲 |
| エッジケース | 境界条件や例外的な入力 |
| レイジー | 必要になるまで実行しない評価戦略。ジェネレーターはレイジーに値を生成する |
| アトミック | 途中で分割できない一体の操作 |
| 副作用 | 関数が戻り値を返す以外に行う外部への影響（ファイル書き込み、グローバル変数の変更等） |
| ハッピーパス | エラーや例外が起きない正常系の処理フロー（happy path） |
| 冪等（べきとう） | 同じ操作を何度実行しても結果が変わらない性質（idempotent） |

### 開発プロセス

| 用語 | 意味 |
|---|---|
| アンチパターン | やってはいけない設計や実装の典型例 |
| デッドコード | 実行されない不要なコード |
| 技術的負債 | 短期的な妥協が将来の開発コストを増大させること |
| フォールバック | 本来の方法が失敗したときの代替手段 |
| フェールオーバー | 障害時に同等の待機系へ自動切替する仕組み |
| フェールバック | フェールオーバー後に主系へ復帰する操作 |
| フォールトトレラント | 一部に障害が起きても全体が停止しない設計思想 |
| 破壊的変更 | 既存のコードが動かなくなる変更（breaking change） |
| ボトルネック | 全体の性能を律速している箇所 |
| スパイク | 技術的な実現可能性を調べるための短期的な試作（spike） |
| ドライラン | 実際の変更を加えずに実行結果だけを確認すること（dry run） |
| リグレッション | 新しい変更によって既存の機能が壊れること（regression） |
| 受け入れ基準 | 機能が完成したと判断するための条件リスト（acceptance criteria） |
| do定義 | 関数やモジュールの責務を「何をするか」で一文で定義すること。エージェントがプラン立案中に使う |
| 明示的コンテキスト | エージェントに曖昧さなく伝えるための具体的な前提情報（explicit context） |

## D-4. 頻出略語

エージェントが展開せずにそのまま使う略語。本書のスコープ（Python開発・テスト・Git・CI/CD）に関連するものを中心に選んだ。

### 設計原則

| 略語 | 正式名称 | 意味 |
|---|---|---|
| DRY | Don't Repeat Yourself | コードの重複を避ける原則 |
| KISS | Keep It Simple, Stupid | シンプルに保つ原則 |
| YAGNI | You Aren't Gonna Need It | 今不要な機能は作らない原則 |
| SOLID | 5つの設計原則の頭文字 | SRP, OCP, LSP, ISP, DIPの総称 |
| SRP | Single Responsibility Principle | 単一責任の原則 |
| DI | Dependency Injection | 依存性注入。テストしやすさに直結する |

### 開発プロセス

| 略語 | 正式名称 | 意味 |
|---|---|---|
| TDD | Test-Driven Development | テスト駆動開発 |
| CI/CD | Continuous Integration / Continuous Deployment | コードの変更を自動でテスト・デプロイする仕組み |
| E2E | End-to-End | 入力から出力まで通しで行うテスト |
| PR | Pull Request | コードの変更をレビュー依頼する仕組み |
| MVP | Minimum Viable Product | 最小限の機能で動く最初のバージョン |
| WIP | Work In Progress | 作業中・未完成 |
| LGTM | Looks Good To Me | レビューOKの意 |

### 技術用語

| 略語 | 正式名称 | 意味 |
|---|---|---|
| API | Application Programming Interface | ソフトウェア間のインターフェース |
| REST | Representational State Transfer | Web APIの設計スタイル |
| CRUD | Create, Read, Update, Delete | データ操作の基本4操作 |
| CLI | Command Line Interface | コマンドラインツール |
| SDK | Software Development Kit | 開発キット |
| ORM | Object-Relational Mapping | Pythonオブジェクトとデータベースの対応付け |
| DTO | Data Transfer Object | データ転送用のオブジェクト |
| EOL | End of Life | サポート終了。Pythonバージョンのサポート期限等で使われる |
