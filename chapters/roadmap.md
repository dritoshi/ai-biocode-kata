## 0. AIコーディングエージェントとの協働（2026年のベストプラクティス）

> 📖 原稿: [00_ai_agent.md](./00_ai_agent.md)

### 0-1. セットアップと基本操作

#### インストール

| | Claude Code CLI | Codex CLI |
|--|----------------|-----------|
| インストール | `npm install -g @anthropic-ai/claude-code` | `npm install -g @openai/codex` |
| 前提 | Node.js 18+ | Node.js 18+（またはbrew/バイナリ） |
| 認証 | Claude Pro/Max契約 or Anthropic APIキー | ChatGPT Plus/Pro契約 or OpenAI APIキー |
| 起動 | `claude` | `codex` |

#### 承認モード — 3段階の安全レベル

両ツールとも「AIにどこまで自由にやらせるか」を3段階で制御する。概念は同じ:

| 安全レベル | 概念 | Claude Code | Codex CLI |
|-----------|------|-------------|-----------|
| 読み取り専用 | コード変更を一切行わず、調査と計画のみ | **Plan Mode** (`Shift+Tab` or `/plan`) | **Read-Only** (`-s read-only`) |
| 承認あり（デフォルト） | 変更のたびに人間が許可 | **Normal Mode** | **Auto**（デフォルト） |
| 全自動 | 編集・実行を自動承認 | **Auto-Accept Mode** (`Shift+Tab`) | `--full-auto` |

**初心者は「承認あり」を基本にすること。** 全自動モードは信頼できるタスクでのみ使う。

#### 基本コマンド

| 操作 | Claude Code | Codex CLI |
|-----|-------------|-----------|
| プロジェクト初期化 | `/init` → CLAUDE.md生成 | AGENTS.mdを手動作成（または `/` コマンドで雛形） |
| 計画モードに切替 | `/plan` or `Shift+Tab`×2 | `-s read-only` / `$create-plan` skill |
| モデル切り替え | `/model` | `/model` |
| セッション再開 | `/resume` | `codex resume` / `codex resume --last` |
| コンテキスト圧縮 | `/compact` | （自動管理） |
| 直前の変更を取り消し | `Esc Esc` (`/rewind`) | git revertで対応 |
| 計画をエディタで編集 | `Ctrl+G` | — |
| 非対話実行 | `claude -p "..."` | `codex exec "..."` |
| ヘルプ | `/help` | `/help` |

### 0-2. Plan → Execute → Review ワークフロー

AIコーディングエージェントを使う上で最も重要な原則は **「いきなり実装させない」** ことである。
Claude Code / Codex CLI ともに推奨される4フェーズワークフロー:

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1: Explore（探索）                                │
│    読み取り専用モードで関連コードを読ませる                │
│    「src/auth/ を読んでセッション管理を理解して」           │
│    Claude Code: Plan Mode / Codex: -s read-only          │
│                                                          │
│  Phase 2: Plan（計画）                                   │
│    読み取り専用モードのまま実装計画を立てさせる             │
│    不明点はエージェントから質問させる                      │
│    Claude Code: Ctrl+G で計画を直接編集                   │
│                                                          │
│  Phase 3: Execute（実装）                                │
│    承認あり or 全自動モードに切り替え                      │
│    計画に沿って実装 → テスト実行 → 修正                   │
│                                                          │
│  Phase 4: Commit（確定）                                 │
│    git diff で変更を確認                                  │
│    テストを実行して全て通ることを確認                      │
│    意味のあるコミットメッセージで記録                      │
└─────────────────────────────────────────────────────────┘
```

**なぜPlan Modeから始めるのか：**
- AIに自由に書かせると「自信満々だが的外れなコード」が生成されることがある
- 計画フェーズは実装フェーズより圧倒的にトークンコストが低い
- 方向修正は計画段階のほうが遥かに安い（実装後のやり直しは高コスト）
- 人間がアーキテクチャの判断をし、AIが実装の詳細を担う、という役割分担が自然に生まれる

**初心者が特に意識すべきこと：**
- 読み取り専用モード中にエージェントが質問してきたら丁寧に答える。この質問への回答が実装品質を大きく左右する
- 計画に納得がいかなければ修正する（Claude Code: `Ctrl+G` で直接編集、Codex: 会話で修正指示）。受け身にならない
- 実装が計画から逸脱したら、一度読み取り専用モードに戻って計画を修正する

#### 「インタビューしてもらう」パターン — 要件が曖昧なときの最強テクニック

自分自身が「何を作りたいか」をうまく言語化できないとき、**エージェントに要件をインタビューさせる**のが非常に有効。
作りたいものの最小限の説明だけ渡して、エージェントに質問攻めにしてもらう。

```
> /plan
> RNA-seqの発現量データからDEG（差次的発現遺伝子）を抽出するCLIツールを作りたい。
> 実装を始める前に、仕様を固めるためにインタビューして。
> 必要な要件を質問して、全体像を理解してから仕様書を作って。
```

エージェントは以下のような質問を投げてくる:
- 入力データの形式は？（カウントマトリクス？ TPM？ どのツールの出力？）
- 比較のデザインは？（2群比較？ 多群？ ペアドサンプル？）
- 統計手法の選択は？（DESeq2ラッパー？ 独自実装？）
- 出力形式は？（TSV？ ボルケーノプロット？ エンリッチメント解析まで？）
- フィルタリング条件は？（FDR閾値、fold change閾値はユーザー指定？）
- 再現性の要件は？（乱数シード固定？ バージョン記録？）

**このパターンが有効な場面：**
- 新しいツールを一から設計するとき
- ドメイン知識はあるがソフトウェア設計の経験が薄いとき（実験系研究者に多い）
- 要件が頭の中にあるが整理されていないとき

**ポイント：**
- 最初のプロンプトは粗くてよい。エージェントの質問に答えていくことで仕様が固まる
- インタビューの結果を仕様書（spec）としてまとめさせる
- 仕様書を確認・修正してから、**新しいセッション**で仕様書を渡して実装を開始する
  - 新しいセッションにする理由: インタビュー過程の大量のやり取りでコンテキストが消費されるため
  - 仕様書がインタビューの「成果物」として、セッション間の橋渡しになる
- 仕様書は `docs/spec.md` としてリポジトリにコミットしておくと、後から設計意図を追える

### 0-3. プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）

プロジェクトのルートに設定ファイルを置くことで、エージェントの振る舞いをカスタマイズできる。

| | Claude Code | Codex CLI |
|--|-------------|-----------|
| ファイル名 | `CLAUDE.md` | `AGENTS.md` |
| 自動生成 | `/init` で生成 | 手動作成（テンプレートを用意すると楽） |
| スコープ | プロジェクト / ユーザー / `~/.claude/` | プロジェクト / ユーザー / `~/.codex/` |

**書くべき内容は同じ。** 以下はバイオインフォプロジェクトの例:

```markdown
# CLAUDE.md / AGENTS.md（共通の内容）

## プロジェクト概要
- RNA-seqパイプラインのPythonラッパーツール
- Python 3.11+, conda環境を使用

## コマンド
- `pytest tests/` — テスト実行
- `ruff check src/` — リント
- `ruff format src/` — フォーマット

## コーディング規約
- 型ヒント必須
- docstringはGoogle style
- テストは `tests/` 配下に `test_*.py` で配置
- ログは `logging` モジュールを使う（`print()` 禁止）

## バイオインフォ固有の注意
- ゲノム座標はBED形式（0-based, half-open）に統一
- BAMファイルの操作は `pysam` を使用
- FASTAのパースは `Bio.SeqIO` を使用（自作パーサは禁止）
```

**設定ファイルのコツ：**
- エージェントが自動で読み取れる情報（コードから推測可能なこと）は書かない — 簡潔に保つ
- 「やってほしいこと」より「やってはいけないこと」の方が効果的
- バイオインフォ特有の規約（座標系、使用ライブラリ）は明示する
- ビルド・テスト・リントのコマンドは必ず書く（エージェントが自分でテスト実行して検証できる）

### 0-4. エージェントにレビューさせる

AIコーディングエージェントは**実装だけでなくレビューにも使える**。
人間のレビューを補完する形で、以下のパターンが有効:

#### 実装後のセルフレビュー
```
# 読み取り専用モードで実装結果をレビューさせる
> 今回の変更をレビューして。以下の観点で問題がないか確認して:
> - エッジケースの見落とし
> - エラーハンドリングの不足
> - パフォーマンス上の懸念
> - 既存テストとの整合性
```

#### 「別人格」レビュー
```
# 新しいセッションを開いて、実装コードを批判的にレビューさせる
> git diffでの変更内容を、シニアエンジニアとしてレビューして。
> バグの可能性、設計上の問題点、テスト不足を厳しく指摘して。
```

#### テスト生成によるレビュー
```
# 実装に対するテストを書かせることで、暗黙の仕様を炙り出す
> src/analysis.py に対するテストを書いて。
> 特にエッジケース（空の入力、不正なFASTAファイル、巨大ファイル）を重視して。
```

**レビューのベストプラクティス：**
- 実装とレビューは**別のセッション**で行うと、コンテキストの偏りが減る
- エージェントのレビュー結果を鵜呑みにしない — 科学的妥当性は人間が判断する
- レビューで指摘された問題は、修正後にテストを追加して再発を防ぐ

### 0-5. サブエージェントとタスク委譲

両ツールとも**サブエージェント**を生成して並列にタスクを実行できる。

| | Claude Code | Codex CLI |
|--|-------------|-----------|
| 自動サブエージェント | Explore Subagent（Plan Mode中にHaikuで自動起動） | — |
| 明示的な委譲 | 「use subagents」と指示 | 「use subagents」と指示 |
| 並列分離 | ワークツリー分離（`isolation: worktree`） | サブエージェントごとに独立実行 |

**初心者が知っておくべきこと：**
- サブエージェントはメインのコンテキストを消費しないため、大規模な調査に向く
- 明示的に「use subagents」「サブエージェントで並列に」と指示すると委譲されやすい
- 各サブエージェントもトークンを消費するため、コスト意識を持つ

### 0-6. コンテキスト管理

AIコーディングエージェントの**最大の制約はコンテキストウィンドウ**（現状200K〜1Mトークン）。
長いセッションでは、初期の指示を「忘れる」ことがある。

**対策：**
- **こまめにコミット** — 1タスク完了ごとにコミット。セッションが壊れても戻れる
- **コンテキスト圧縮** — Claude Code: `/compact`、Codex: 自動管理
- **セッションの切り分け** — 大きなタスクは複数セッションに分割
- **プロジェクト設定ファイル** — セッションをまたいで保持すべき情報は CLAUDE.md / AGENTS.md に書く
- **セッション再開** — Claude Code: `/resume`、Codex: `codex resume`

### 0-7. モデル選択・推論の深さ・コスト意識

AIコーディングエージェントの性能を調整するには **「どのモデルを使うか」** と **「どれだけ深く考えさせるか」** の2つの軸がある。

#### モデルの階層

| 役割 | Claude Code | Codex CLI |
|-----|-------------|-----------|
| 最高精度（複雑な設計判断、レビュー） | **Opus** | **GPT-5.4** |
| バランス（日常の実装、デフォルト） | **Sonnet** | **GPT-5.3-Codex** |
| 高速・低コスト（探索、簡単な修正） | **Haiku** | **GPT-5.3-Codex-Spark** |
| 切り替え方法 | `/model` | `/model` |

#### 推論の深さ（Reasoning Depth）— 2つ目の調整軸

モデル選択とは**独立して**、エージェントにどれだけ「考えさせるか」を調整できる。
推論を深くするほど精度が上がるが、レイテンシとコストが増える。

| | Claude Code | Codex CLI |
|--|-------------|-----------|
| 概念 | **Extended Thinking** | **Reasoning Effort** |
| 段階 | ON / OFF（Opus 4.6ではadaptive thinkingが自動調節） | None / Low / Medium / High / Extra High |
| 切り替え方法 | `Alt+T`（macOS: `Option+T`）でトグル | `/model` → Reasoning effort、または`-c model_reasoning_effort="high"` |
| 計画時の設定 | Plan Mode + Extended Thinking の組み合わせ | `plan_mode_reasoning_effort` で計画時だけ別の深さを設定可能 |
| 設定の永続化 | — | `~/.codex/config.toml` に `model_reasoning_effort = "high"` |

#### 使い分けの指針

```
簡単な修正（タイポ修正、1行変更）
  → 軽量モデル + 低い推論深度
    Claude Code: Sonnet（adaptive thinkingに任せる）
    Codex: GPT-5.3-Codex, Reasoning = Low

日常的な実装（関数追加、テスト生成）
  → デフォルトモデル + デフォルト推論深度
    Claude Code: Sonnet
    Codex: GPT-5.3-Codex, Reasoning = Medium（デフォルト）

複雑な設計判断（アーキテクチャ、リファクタリング計画）
  → 高精度モデル + 高い推論深度
    Claude Code: Opus + Extended Thinking ON
    Codex: GPT-5.4, Reasoning = High / Extra High
```

**コスト意識：**
- サブスクリプション契約（Claude Max, ChatGPT Pro等）はトークン上限なし/大
- API利用は従量課金 — 高精度モデル×高推論深度は5〜10倍のコスト
- 「全てに最高性能を使う」のではなく、タスクに応じて使い分ける

### 0-8. AIに頼るべきこと・自分で判断すべきこと

| AIが得意なこと | 人間が判断すべきこと |
|--------------|-------------------|
| 定型コードの生成 | アーキテクチャ・設計方針 |
| APIの使い方・ライブラリの呼び出し | データ構造・フォーマットの選択 |
| エラーメッセージの解読 | 科学的妥当性の検証 |
| テストコードの生成 | テストすべきケースの選定 |
| リファクタリング | 分割の粒度と命名の判断 |
| ドキュメントの下書き | 内容の正確性の確認 |
| SLURMスクリプトの雛形 | リソース要求量の見積もり |

**「動くコード」と「良いコード」の違い：**
- AIが生成したコードは「動く」ことが多いが、それが「良い」とは限らない
- 不要な複雑さ、非効率なアルゴリズム、ハードコーディングされた設定を見抜く目が必要
- レビュー（§0-4）とテスト（§7）で品質を担保する

### 0-9. 車輪の再発明を防ぐ

- エージェントに「この処理を実装して」と頼む前に、既存ツール/ライブラリがないか確認する
- **判断基準**: 既存ツールが成熟・テスト済みなら呼ぶ。学習目的やカスタマイズが必要なら実装する
- プロジェクト設定ファイルに「FASTAパーサは `Bio.SeqIO` を使え、自作禁止」のように明記する
- 「この処理を実装する前に、既存のPythonライブラリやCLIツールで解決できないか確認して」と聞く習慣をつける

### 0-10. セキュリティと権限管理

| | Claude Code | Codex CLI |
|--|-------------|-----------|
| 権限ホワイトリスト | `/permissions`（例: `Bash(pytest *)`, `Edit(src/**)`） | サンドボックス設定（`sandbox_mode`） |

## 1. 設計原則

> 📖 原稿: [01_design.md](./01_design.md)

### 1-1. コードの設計原則
- **KISS原則**（Keep It Simple, Stupid）— 不必要な複雑さを避ける
- **DRY原則**（Don't Repeat Yourself）— 重複コードの排除と関数化
- **YAGNI原則**（You Aren't Gonna Need It）— 必要になるまで作らない
- **関心の分離** — 機能ごとにモジュールを分ける
- **単一責任原則** — 1つの関数は1つのことだけ行う
- **最小驚き原則** — 予測可能な振る舞いをする

### 1-2. UNIX哲学
- 「一つのことをうまくやるプログラムを書け」
- テキストストリームをユニバーサルインターフェースとする
- パイプ（`|`）とリダイレクション（`>`, `>>`, `<`, `2>&1`）
- stdin / stdout / stderr の使い分け
- 小さなツールの組み合わせで複雑な処理を実現
- 終了ステータス（exit code: 0 = 成功、非0 = 失敗）

### 1-3. ソフトウェア開発手法の用語 — AIエージェントと会話するための語彙

AIコーディングエージェントに計画を立てさせると「スプリントで分割」「MVPから始める」「リファクタリングのイテレーション」等の用語が当然のように出てくる。
これらはソフトウェア業界で広く使われている開発手法の用語であり、意味を知らないとAIの提案を正しく評価できない。

#### 開発プロセスモデル

**ウォーターフォール** — 要件定義→設計→実装→テスト→リリースを順番に進める古典的モデル。前のフェーズに戻りにくい。
大規模な受託開発で使われてきたが、研究開発のように要件が変わりやすいプロジェクトには不向き。

**アジャイル開発** — 小さな単位で「動くもの」を繰り返し作りながら改善する考え方。ウォーターフォールの対極。
以下のスクラムやカンバンはアジャイルの具体的な実践手法。
研究のプログラミングは本質的にアジャイル的（要件が進行中に変わる）。

**スクラム** — アジャイルの代表的なフレームワーク。以下の用語が頻出する:
- **スプリント** — 1〜2週間の固定期間の開発サイクル。「この機能をこのスプリントで完成させる」
- **バックログ** — やるべきタスクの優先順位つきリスト。GitHub Issuesで管理できる
- **スタンドアップ** — 毎日の短い進捗共有（1人開発では不要だが、用語として知っておく）
- **レトロスペクティブ** — スプリント終了後の振り返り。何がうまくいき、何を改善すべきか

**カンバン** — タスクを「To Do → In Progress → Done」のボード上で視覚管理する手法。
GitHub ProjectsやNotionのボードビューがこれ。進行中のタスク数に上限（WIP制限）を設けるのが特徴。

#### よく出てくる開発プラクティスの用語

| 用語 | 意味 | AIエージェントとの関連 |
|-----|------|-------------------|
| **MVP**（Minimum Viable Product） | 最小限の動くプロダクト。まず核心機能だけ作り、後から拡張する | エージェントが「まずMVPを作りましょう」と提案してきたら、全機能を一度に実装しない戦略のこと |
| **イテレーション** | 「計画→実装→評価」を繰り返すこと | Plan→Execute→Reviewサイクル（§0-2）そのもの |
| **リファクタリング** | 動作を変えずにコードの内部構造を改善する | 「テストを通してからリファクタリング」が鉄則 |
| **技術的負債**（テクニカルデット） | 急いで書いたコードが将来の開発を遅くする「借金」 | エージェントが「技術的負債になる」と警告したら、設計を見直すサイン |
| **スコープクリープ** | 開発中に要件がどんどん膨らむこと | エージェントに「あれもこれも」と追加依頼し続けると発生する |
| **ブロッカー** | 他の作業を止めてしまう障害 | 依存ライブラリのインストール失敗など。解決するまで先に進めない |
| **デプロイ** | ソフトウェアを実際に使える環境に配置すること | HPCにツールを入れて使えるようにする = デプロイ |
| **CI/CD** | 継続的インテグレーション/デリバリー | コミットするたびにテストが自動実行される仕組み（§7-4） |
| **コードレビュー** | 他の人（やAI）にコードを読んでもらい問題を見つける | §0-4 のレビューワークフロー |
| **ペアプログラミング** | 2人で1つのコードを書く手法 | AIエージェントとの対話は一種のペアプログラミング |
| **プルリクエスト（PR）** | 変更を本体に統合する前のレビュー依頼 | エージェントに `gh pr create` でPRを作らせられる |
| **マイルストーン** | 大きな目標に向けた中間の達成地点 | 「Phase 1のマイルストーン: FASTQの読み込みとQCが動く」 |

#### 一人開発で実際に使うもの

上記の全てを厳密に実践する必要はない。一人で研究ツールを開発する場合に特に有用なのは:

- **MVPの考え方** — 最初から完璧を目指さない。まず最小限動くものを作り、使いながら改善する
- **イテレーション** — Plan→Execute→Reviewの繰り返し（§0-2のワークフローそのもの）
- **バックログ（≒ GitHub Issues）** — やりたいことをIssueに書き出して優先順位をつける
- **リファクタリング** — 「動いたけど汚いコード」をテストで保護してから整理する
- **技術的負債の意識** — 「今は動くけど後で困る」コードを放置しない

エージェントが「スプリントで区切りましょう」と提案してきたら、
それは「1-2週間で達成可能な小さなゴールを設定して段階的に進めよう」という意味。
大げさに構える必要はなく、研究のマイルストーン設計と同じ感覚で理解すればよい。

## 3. 計算機科学の基礎知識

### 3-1. データ構造と計算量
- list, dict, set, tuple, deque の使い分け
- O記法（O(1), O(n), O(n log n), O(n²)）
- 「なぜこのデータ構造を選ぶのか」を説明できること
  - 例: 10万遺伝子のID検索 → list の `in` はO(n)、set の `in` はO(1)

### 3-2. 数値表現と浮動小数点
- 整数型のビット幅（int8, int32, int64）とオーバーフロー
- **IEEE 754 浮動小数点** — float32 / float64 の精度の限界
  - **`0.1 + 0.2 != 0.3` 問題** — なぜ起きるか、どう対処するか
- **丸め誤差の蓄積** — `math.fsum()` の利用
- **比較の注意** — `==` ではなく `math.isclose()` / `np.allclose()` を使う
- 特殊な値 — `inf`, `NaN` の振る舞い（`NaN != NaN`）

> **🧬 コラム：バイオインフォで踏む浮動小数点の罠**
>
> - 正規化値（TPM, FPKM）の比較で `==` を使うと一致しない
> - p値の極小値（`1e-300`付近）→ アンダーフローで `0.0` になる → 対数p値で扱う
> - GPU計算（float32）で発現量行列の精度が落ちる → 正規化後の微小差が消失
> - Phredスコア: Q30 = 10^-3 の対数変換、精度に注意

### 3-3. 文字エンコーディング
- ASCII, UTF-8 の基礎
- **改行コードの罠** — LF（Unix）vs CR+LF（Windows）
  - Windows由来のFASTAファイルが壊れるパターン → `dos2unix`
- バイトオーダー（エンディアン）— バイナリフォーマットで意識が必要

### 3-4. 乱数と再現性
- 擬似乱数生成器（PRNG）と乱数シードの固定
- `np.random.default_rng(seed)` で再現性を確保
- 論文にシード値を記録する習慣

> **🧬 コラム：バイオインフォでの乱数再現性**
>
> - t-SNE / UMAP の結果が毎回変わる → シード固定で解決
> - GPU（cuDNN）の非決定的演算 → `torch.use_deterministic_algorithms(True)` 等
> - ブートストラップ法やランダムサブサンプリングにもシード記録が必須

### 3-5. 計算機アーキテクチャの基礎
- メモリ階層（キャッシュ → RAM → ストレージ）
- 行優先（C order）vs 列優先（Fortran order）— NumPy配列のメモリレイアウト
- CPU vs GPU の使い分け、VRAM の制約
- I/Oバウンド vs コンピュートバウンドの見極め
- ネットワークファイルシステム（NFS, Lustre）の特性

> **🧬 コラム：バイオインフォの主要フォーマット**
>
> 自分でパーサを書く前に、各フォーマットの標準を知ること。
>
> | カテゴリ | 形式 | 注意点 |
> |---------|------|--------|
> | 配列 | FASTA, FASTQ | FASTAインデックス（`.fai`）でランダムアクセス可能 |
> | マッピング | SAM/BAM/CRAM | BAM+BAIが日常使い。FLAGフィールドはビットフラグ |
> | アノテーション | BED, GFF3/GTF | **BED=0-based**, **GFF=1-based** — 混同は致命的バグの元 |
> | 変異 | VCF/BCF | ヘッダ行（`##`）の構造を理解する |
> | MSA | FASTA, Clustal, Stockholm, PHYLIP | ツール間の変換は `Biopython AlignIO` or EMBOSS `seqret` |
> | BLAST出力 | outfmt 6/7（タブ区切り） | カラムはカスタマイズ可能。XML (outfmt 5) はプログラムパース向け |
> | シグナル | BigWig, BedGraph | ゲノムカバレッジ・ChIPシグナル等 |
> | シングルセル | AnnData (.h5ad) | `.X`（行列）, `.obs`（細胞）, `.var`（遺伝子） |
> | 系統樹 | Newick, Nexus | 括弧表記 `((A,B),C);` |
>
> 座標系の違い（0-based half-open vs 1-based closed）は最も多いバグ源の一つ。

## 8. 成果物の形式とプロジェクト設計

開発を始める前に**「最終的にどういう形で提供するのか」**を決める。
これを決めないまま AIエージェントに「作って」と言うと、的外れなプロジェクト構成が生成される。
CLAUDE.md / AGENTS.md に「このプロジェクトはpipインストール可能なライブラリとして公開する」と明記するだけで、
生成されるコードの構造が大きく変わる。

### 8-1. 成果物の形式の選択

以下のパターンから、自分の目的に合ったものを選ぶ。
複数を組み合わせることもある（例: ライブラリ＋CLIエントリポイント）。

#### パターン1: スタンドアロンスクリプト

**概要:** 1つの `.py` ファイルで完結する。最も手軽。

**向いている場面:**
- 一回きりの解析、使い捨てスクリプト
- 自分しか使わない前処理・変換スクリプト
- 数百行以内の小さな処理

**構成例:**
```
filter_vcf.py          # スクリプト本体
README.md              # 使い方
```

**実行方法:** `python filter_vcf.py --input variants.vcf --min-qual 30`

**プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）への記述例:**
```
このプロジェクトは単一のPythonスクリプト（filter_vcf.py）。
パッケージ化はしない。argparseでCLIを提供する。
```

**注意:** 大きくなったらライブラリ化を検討する。500行を超えたらほぼ確実に分割すべき。

#### パターン2: pipインストール可能なPythonパッケージ

**概要:** `pip install` でインストールでき、`import mypackage` で使えるライブラリ。
CLIエントリポイントも同時に提供できる。

**向いている場面:**
- 他の人にも使ってほしいツール
- 複数のモジュールに分かれる中〜大規模なプロジェクト
- PyPIで公開したい場合

**構成例:**
```
my_tool/
├── src/my_tool/
│   ├── __init__.py
│   ├── io.py
│   ├── analysis.py
│   ├── cli.py           # CLIエントリポイント
│   └── viz.py
├── tests/
├── pyproject.toml        # ★ これが核心
├── README.md
├── LICENSE
└── .gitignore
```

**核心となる `pyproject.toml`:**
```toml
[project]
name = "my-tool"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = [
    "biopython>=1.80",
    "pysam>=0.22",
    "click>=8.0",
]

[project.scripts]
my-tool = "my_tool.cli:main"    # CLIコマンドの登録
```

**インストールと実行:**
```bash
pip install .                     # ローカルインストール
pip install -e .                  # 開発モード（コード変更が即反映）
my-tool align --input reads.fq   # CLIとして実行
```

```python
from my_tool.analysis import run_analysis   # ライブラリとして利用
```

**公開:**
```bash
pip install build twine
python -m build                   # sdist + wheel を生成
twine upload dist/*               # PyPIにアップロード
```

**プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）への記述例:**
```
このプロジェクトはpipインストール可能なPythonパッケージ。
src layoutを使う。CLIエントリポイントはclickで実装。
pyproject.tomlで依存関係を管理する。
```

#### パターン3: condaパッケージ

**概要:** `conda install` でインストールできる形式。バイオインフォではbiocondaチャネルで公開するのがデファクト。

**向いている場面:**
- C/C++の依存（htslib等）がありpipだけでは解決しにくいとき
- biocondaエコシステムの一部として公開したいとき
- conda環境での再現性を重視するとき

**追加で必要なもの:**
- `meta.yaml`（condaビルドレシピ）
- biocondaへのPR（bioconda-recipesリポジトリ）

**実装ステップ:**
1. まずpipインストール可能なパッケージ（パターン2）を完成させる
2. PyPIに公開する
3. `meta.yaml` を書いてbioconda-recipesにPRを出す

**注意:** condaパッケージ化はパターン2の上に成り立つ。いきなりcondaだけを目指さない。

#### パターン4: シェルスクリプトで繋いだパイプライン

**概要:** 複数のコマンド（既存ツール＋自作スクリプト）をシェルスクリプトでつなぐ。

**向いている場面:**
- 既存ツール（samtools, STAR, featureCounts等）を順番に実行する定型パイプライン
- 手動で毎回コマンドを打つのが面倒になったとき
- ワークフロー言語を導入するほど複雑でない場合

**構成例:**
```
pipeline/
├── run_rnaseq.sh         # メインのパイプラインスクリプト
├── scripts/
│   ├── filter_lowqual.py # 自作の前処理スクリプト
│   └── summarize.py      # 結果集計
├── config.sh             # パラメータ設定
└── README.md
```

**実装のコツ:**
- `set -euo pipefail` を必ず書く（エラー時に即停止）
- パラメータは `config.sh` で外部化する
- ログをファイルに保存する（`2>&1 | tee log.txt`）
- 各ステップの成功/失敗をチェックする

**プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）への記述例:**
```
このプロジェクトはBashパイプライン。scripts/にPython補助スクリプトを置く。
config.shでパラメータを管理する。set -euo pipefail必須。
```

**限界:** サンプル数が多い場合や条件分岐が複雑な場合はワークフロー言語（パターン5）に移行する。

#### パターン5: ワークフロー言語（Snakemake / Nextflow）

**概要:** 依存関係・並列実行・再開機能を持つ本格的なパイプライン。

**向いている場面:**
- 多サンプル処理（10+サンプル）
- ステップ間の依存関係が複雑
- HPC上での並列実行が必要
- 中間結果からの再開が必要
- 他の研究者が再現実行する想定

**構成例（Snakemake）:**
```
workflow/
├── Snakefile              # ワークフロー定義
├── config/
│   └── config.yaml        # パラメータ
├── rules/
│   ├── mapping.smk        # マッピングルール
│   ├── quantification.smk # 定量ルール
│   └── qc.smk            # QCルール
├── scripts/
│   ├── deseq2.py         # カスタム解析スクリプト
│   └── plot_volcano.py
├── envs/
│   └── mapping.yaml       # conda環境定義
└── README.md
```

**プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）への記述例:**
```
このプロジェクトはSnakemakeワークフロー。
ルールはrules/に分割する。各ルールにconda環境を指定する。
configはconfig/config.yamlで管理する。
```

**パターン4からの移行サイン:**
- サンプルごとにコマンドをコピペしている → ワイルドカードで解決
- 途中で失敗したら最初からやり直し → `--rerun-incomplete` で途中再開
- HPCのジョブ管理が手動 → `--cluster`/`--slurm` で自動化

#### パターン8: Webアプリ（Streamlit / Gradio）

**概要:** 非プログラマーでもブラウザから使えるGUIを提供する。

**向いている場面:**
- 共同研究者（実験系）に解析ツールを使ってもらいたいとき
- パラメータを変えながらインタラクティブに結果を確認したいとき

**構成例:**
```
my_app/
├── app.py                # Streamlitアプリ本体
├── src/my_tool/          # ロジック（パターン2と同じ）
│   ├── analysis.py
│   └── viz.py
├── pyproject.toml
├── requirements.txt
└── README.md
```

**実装のコツ:**
- **ロジックとUIを分離する** — `app.py` はGlueコードのみ。解析ロジックは `src/` に置く

- Gradioは機械学習モデルのデモに特に適している
- HPC上で動かす場合はポートフォワーディングが必要（§2-5参照）

**プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）への記述例:**

このプロジェクトはStreamlitアプリ。app.pyにUIコードを書き、
解析ロジックはsrc/my_tool/に分離する。UIとロジックを混ぜない。
```

**注意:** Webアプリは「届け方」であり「作り方」ではない。
まずパターン2でロジックを作り、それをStreamlitで包む、という順序で進める。

#### パターン9: コンパイル済みバイナリ

**概要:** Pythonがインストールされていない環境でも動く実行ファイルとして配布する。

**向いている場面:**
- 配布先にPython環境がない（稀だがある）
- ユーザーにpipやcondaの操作を求めたくないとき
- GitHub Releasesでワンバイナリを配布したいとき

**ツール:**
- **PyInstaller** — 最も広く使われる。`pyinstaller --onefile my_tool.py` で単一バイナリ生成
- **Nuitka** — PythonコードをCにコンパイル。PyInstallerより高速な実行ファイルを生成

**注意:**
- バイナリサイズが大きくなりがち（数十〜数百MB）

- GitHub Actionsでクロスプラットフォームビルドを自動化するのがベストプラクティス
- **バイオインフォでは稀なパターン** — 多くの場合condaやコンテナのほうが適切

**プロジェクト設定ファイル（CLAUDE.md / AGENTS.md）への記述例:**
```
このプロジェクトはPyInstallerで単一バイナリとして配布する。
GitHub Actionsでlinux/macOSのバイナリを自動ビルドしてReleasesに添付する。
```

### 8-2. どのパターンを選ぶか — 判断フローチャート

```
自分だけが使う一回きりの処理？
  → Yes: パターン1（スクリプト）
  → No ↓

既存ツールを順番に実行するだけ？
  → Yes: サンプル数10以下 → パターン4（シェルスクリプト）
         サンプル数10以上 or HPC → パターン5（ワークフロー言語）
  → No ↓

Pythonで独自のロジックを実装する？
  → Yes: 他の人にも使ってほしい → パターン2（pipパッケージ）
         自分のラボ内で使う    → パターン2（軽量版）
  → No ↓

非プログラマーにGUIで使わせたい？
  → Yes: パターン8（Streamlit）+ ロジックはパターン2に分離

解析結果のレポートを共有したい？
  → Yes: パターン7（Notebook）+ ロジックはパターン2に分離
```

**よくある組み合わせ:**
- **ライブラリ＋CLI＋コンテナ**: パターン2 + パターン6（最も汎用的）
- **ワークフロー＋コンテナ**: パターン5 + パターン6（論文再現性に最適）
- **ライブラリ＋Streamlit**: パターン2 + パターン8（共同研究者向け）
- **ライブラリ＋conda**: パターン2 + パターン3（biocondaエコシステム統合）

**エージェントへの最初の指示に含めるべきこと:**
```
> /plan
> FASTQファイルのクオリティフィルタリングツールを作りたい。
> 成果物の形式: pipインストール可能なPythonパッケージ（CLIつき）
> 対象ユーザー: ラボのメンバー
> インタビューして仕様を固めて。
```

「成果物の形式」を最初に伝えることで、エージェントは適切なプロジェクト構成、
`pyproject.toml`、エントリポイント設計を最初から考慮した計画を立てる。

- ハードコーディングを避ける
- 設定ファイル（YAML / TOML）による外部化

- 環境変数による秘密情報の管理
- デフォルト値の設計

### 8-5. エラーハンドリング
- 例外の使い分け（ValueError, FileNotFoundError, カスタム例外）
- 早期リターン（ガード節）パターン
- ユーザーに分かりやすいエラーメッセージ
- fail-fast の原則

### 8-6. 段階的な成長パス — スクリプトから育てる

最初から完璧な形式を目指す必要はない。
多くの研究ツールは以下の段階を経て成熟する:

```
パターン1                 パターン2               パターン2+3
スクリプト ───────→ pip パッケージ ───────→ conda/PyPI公開
 (自分用)           (CLIつき)              (コミュニティへ)
   │                   │                       │
   │                   ├─→ パターン5            ├─→ パターン6
   │                   │   ワークフロー統合       │   コンテナ化
   │                   │                       │
   │                   └─→ パターン8            └─→ 論文公開
   │                       Streamlit GUI            (再現性保証)
   │
   └─→ パターン4
       シェルパイプライン ──→ パターン5
                            ワークフロー言語
```

**成長のタイミングの目安:**
- スクリプトが500行を超えた → パッケージ化を検討
- `python script.py` を何度もコピペしている → CLIエントリポイントを作る
- 「あのスクリプトどこだっけ」が頻発 → `pip install -e .` で開発モード
- 共同研究者から「使い方がわからない」 → README + Streamlit GUI
- 論文投稿時 → Dockerfile + ワークフロー + バージョン固定
- 他のラボから「使いたい」 → PyPI/bioconda公開

**AIエージェントとの連携:**
```
> /plan
> このスクリプト（filter_vcf.py, 600行）をpipインストール可能なパッケージに
> リファクタリングしたい。現在の機能を壊さずに段階的に移行する計画を立てて。
```
エージェントはテストを書きながら段階的にリファクタリングする計画を立ててくれる。
一気に書き換えるより、テストで保護しながら少しずつ移行するのが安全。

## 10. データ処理ライブラリ

### 10-1. 科学計算の基盤
- **NumPy** — 配列演算、ブロードキャスティング
- **pandas** — データフレーム操作
- **SciPy** — 統計検定、距離計算
- **polars** — 大規模データの高速処理

> **🧬 コラム：バイオインフォマティクスのPythonライブラリ**
>
> | ライブラリ | 用途 | 代表的な使い方 |
> |-----------|------|---------------|
> | **Biopython** | 配列操作の万能ナイフ | `SeqIO`（FASTA/FASTQ）, `AlignIO`（MSA）, `Blast`（結果パース）, `Entrez`（NCBI API） |
> | **pysam** | SAM/BAM/VCF操作 | htslib のPythonラッパー。高速 |
> | **pyBigWig** | BigWigの読み書き | ゲノムシグナルデータ |
> | **scanpy** | シングルセル解析 | 前処理→クラスタリング→可視化の一気通貫 |
> | **anndata** | シングルセルデータ構造 | `.X`, `.obs`, `.var`, `.obsm` |
> | **pyranges** | ゲノム区間演算 | BED的操作をpandas風に |
> | **ETE Toolkit** | 系統樹操作・可視化 | 系統樹の読み込み、装飾、解析 |
>
> エージェントに実装を頼む前に、これらのライブラリが提供する機能を確認する。
> 例: FASTAのパーサを自作する必要はない → `Bio.SeqIO.parse()` を使う。

> **🤖 コラム：機械学習ライブラリ**
>
> バイオインフォ×MLで頻出するライブラリ群。配列解析と同じく車輪の再発明を避ける。
>
> | ライブラリ | 用途 | 代表的な使い方 |
> |-----------|------|---------------|
> | **scikit-learn** | 古典的ML | 分類、クラスタリング、前処理パイプライン、交差検証 |
> | **PyTorch** | 深層学習 | カスタムモデル定義、学習ループ、GPU計算 |
> | **JAX** | 高速数値計算 | 自動微分、JITコンパイル、関数型スタイル |
> | **Hugging Face** | 事前学習モデル | Transformers, Datasets, tokenizers。DNAモデル（DNABERT等）もここにある |
> | **Lightning** | 学習ループの定型化 | PyTorch Lightningでボイラープレート削減 |
> | **wandb** | 実験追跡 | 学習曲線、ハイパーパラメータ、モデルチェックポイント（§13A参照） |
> | **optuna** | ハイパラ最適化 | ベイズ最適化ベースの自動チューニング |
>
> 配列解析からMLに踏み出す典型的な流れ:
> 1. scikit-learnで特徴量ベースの古典的MLを試す（まずベースラインを作る）
> 2. PyTorchでカスタムモデルを構築
> 3. Hugging Faceで事前学習モデルのfine-tuning

## 13A. 実験管理（ML/計算実験の追跡）

> 🤖 この章はバイオインフォ×機械学習プロジェクト向けの内容

### 13A-1. なぜ実験管理が必要か
- 「先週のあの結果、どのパラメータで出したんだっけ？」を防ぐ
- ハイパーパラメータ、メトリクス、モデルチェックポイントの一元管理
- 配列解析でも：パイプラインのパラメータ違い（閾値、リファレンス版）の比較に有用

### 13A-2. 実験追跡ツール

| ツール | 特徴 | 向いている場面 |
|-------|------|---------------|
| **wandb** (Weights & Biases) | クラウドベース、可視化が美しい、チーム共有が容易 | MLの学習実験、ハイパラ探索 |
| **MLflow** | OSS、ローカル/サーバー両対応、モデルレジストリ | モデル管理、デプロイ連携 |
| **hydra** | 設定ファイルの構成管理（実験追跡そのものではない） | パラメータの組み合わせ管理 |
| **DVC** (Data Version Control) | データとモデルのバージョン管理（Git連携） | 大規模データのバージョニング |

### 13A-3. 最小限の実験管理パターン

ML専用ツールを導入する前に、まずこれだけやる:

```python
import json, datetime, hashlib

def log_experiment(params: dict, metrics: dict, output_dir: str):
    """最小限の実験ログ"""
    record = {
        "timestamp": datetime.datetime.now().isoformat(),
        "params": params,
        "metrics": metrics,
        "git_hash": os.popen("git rev-parse HEAD").read().strip(),
    }
    path = f"{output_dir}/experiment_log.jsonl"
    with open(path, "a") as f:
        f.write(json.dumps(record) + "\n")
```

**段階的な導入パス:**
1. まず上のような自前JSONLログを書く習慣をつける
2. 学習実験が増えたら **wandb** を導入（`wandb.init()`, `wandb.log()`）
3. パラメータの組み合わせが爆発したら **hydra** で設定管理
4. データセット自体のバージョン管理が必要なら **DVC**

## 14. HPC

### 14-1. HPCの基本
- ログインノード vs 計算ノード（ログインノードで重い処理をしない）
- キュー / パーティション
- リソース申請（ノード、CPU、メモリ、GPU）

### 14-2. SLURM
- `sbatch`, `squeue`, `scancel`, `sinfo`, `sacct`
- ジョブスクリプト（`#SBATCH` ディレクティブ）
- アレイジョブ（`--array`）、依存ジョブ（`--dependency`）
- GPU申請（`--gres=gpu:1`）
- インタラクティブジョブ（`srun --pty bash`）

### 14-3. HPCでのデータ管理
- ホーム vs スクラッチの使い分け
- モジュールシステム（`module load`）
- 共有ファイルシステムの注意（大量の小ファイルを避ける）

> **🤖 コラム：GPU学習ジョブのSLURMスクリプト**
>
> ```bash
> #!/bin/bash
> #SBATCH --job-name=train_model
> #SBATCH --partition=gpu
> #SBATCH --gres=gpu:1              # GPU 1枚
> #SBATCH --cpus-per-task=8         # データローダー用のCPU
> #SBATCH --mem=32G
> #SBATCH --time=24:00:00
> #SBATCH --output=logs/%x_%j.out
> #SBATCH --error=logs/%x_%j.err
>
> module load cuda/12.1
> source activate ml-env
>
> # GPU状態の記録（デバッグ用）
> nvidia-smi
>
> python train.py \
>     --epochs 100 \
>     --batch-size 64 \
>     --learning-rate 1e-4 \
>     --wandb-project my-project     # 実験追跡（§13A）
> ```
>
> **よくあるポイント:**
> - `--cpus-per-task` はデータローダーの `num_workers` に合わせる
> - `nvidia-smi` をジョブ冒頭で呼んでGPUの認識を確認する習慣
> - OOM（GPU メモリ不足）→ バッチサイズを下げる or gradient accumulation
> - マルチGPUは `--gres=gpu:2` + PyTorch DDP（上級者向け）
