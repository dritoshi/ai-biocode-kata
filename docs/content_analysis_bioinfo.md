# バイオインフォマティクスカリキュラム比較分析

## 分析の枠組み

### 目的

バイオインフォマティクス特化のカリキュラム・教材と本書を比較し、AIエージェントとの協働（Vibe coding）に必要な「概念的語彙・判断力」の不足を特定する。

**重要な制約**: バイオインフォの手法そのものを教えることが目的ではなく、AIエージェントに正確な指示を出し、生成コードをレビューするための知識が対象。

### 3層フィルター

各カリキュラムから抽出したトピックに以下のフィルターを適用し、本書スコープ内かを判定した:

| フィルター | 判断基準 | 対象例 | 対象外の例 |
|---|---|---|---|
| **指示力** | この概念を知っているとAIへの指示が具体的になるか | 「RNA-seqにはSTAR、ロングリードにはminimap2」と指定できる | 動的計画法の実装 |
| **レビュー力** | AI生成コードの品質を評価できるか | 正規化手法（CPM/TPM/FPKM）の妥当性をチェックできる | HMMのパラメータ推定 |
| **判断力** | 解析の設計判断ができるか | リファレンスゲノムの選択、リプリケートの種別 | de Bruijnグラフの構築 |

## B. 不足の可能性があるトピック

### 高優先度

#### 1. QCメトリクスの解釈

**現状:** §12のRNA-seqパイプラインにFastQCステップが含まれ、§5のツール一覧にfastpがあるが、QCレポートの読み方・判断基準がない。

**なぜ不足か（3層フィルター）:**
- **レビュー力**: AIがQCステップを含むパイプラインを生成した際、QCの結果が「合格」か「要調査」かを判断できない
- **判断力**: マッピング率が70%のとき、それが正常なのか問題なのかを判断できない（生物種・実験種別で異なる）
- **指示力**: 「マッピング率が低い場合はアダプターのトリミングを追加して」のような条件付き指示が書けない

**カリキュラムでの扱い:**
- EMBL-EBI: RNA-seq QCを明示的に扱う（QC → マッピング → 定量のワークフロー）
- CBW: 各オミクス解析ワークショップでQCが冒頭に含まれる
- GTN: Sequence Analysis の Quality Control チュートリアルあり

**本書で不足しているQCメトリクス:**
| メトリクス | 意味 | 判断の目安 | 影響する判断 |
|---|---|---|---|
| Per base quality (Phred) | 各ポジションの塩基品質 | Q30以上が目安 | トリミング範囲の決定 |
| GC content分布 | GC含量のばらつき | 理論分布からの乖離 = 汚染の兆候 | コンタミネーション調査の要否 |
| Duplication rate | PCR重複の割合 | ライブラリ複雑度の指標 | dedup処理の要否 |
| Adapter content | アダプター配列の残留 | >5%なら要トリミング | fastp/Trimmomatic設定 |
| Mapping rate | リードのマッピング率 | 生物種・手法依存（RNA-seq: >70-80%） | リファレンスの確認、汚染チェック |
| Uniquely mapped rate | ユニークマッピングの割合 | マルチマッピングの多さ = 反復配列 | カウント方法の選択 |

**推奨:** §12-1「パイプラインの構成要素」の中に「QCメトリクスの読み方」サブセクション（1ページ程度）を追加。または§4-2の「フォーマット選択」直後に🧬コラムとして配置。目安の数値は「生物種・実験手法で異なるため参考値」と明記し、AIに「このサンプルのマッピング率は75%だが、ヒトRNA-seqとしては正常か確認して」と聞けるようになることをゴールとする。

#### 2. 実験デザインの基礎概念（リプリケート・バッチ効果・交絡因子）

**現状:** 本書全体を通じて、リプリケートの種別、バッチ効果、交絡因子、サンプルサイズの議論がない。

**なぜ不足か（3層フィルター）:**
- **判断力**: AIにDEG解析を依頼するとき「リプリケートは3つ」「バッチ情報をモデルに含めるべきか」を判断できない
- **レビュー力**: AIが2群比較のコードを生成したとき、バッチ効果の補正が含まれていなくても気づけない
- **指示力**: 「バッチ変数を共変量として含めて」「テクニカルリプリケートは平均してからバイオロジカルリプリケート間で検定して」と指示できない

**カリキュラムでの扱い:**
- Harvard MCB112: 統計検定とP値を深く扱い、計算的陽性/陰性対照実験を重視
- EMBL-EBI: Functional Genomics IIでexperimental design（replicates, library preparation）を明示的に扱う
- CBW: 各解析ワークショップでデータの実験デザインが前提知識として扱われる
- Buffalo: 再現可能な研究の文脈でプロジェクト設計を扱う

**本書で不足している概念:**
| 概念 | 定義 | Vibe codingでの関連場面 |
|---|---|---|
| **バイオロジカルリプリケート** | 独立した生物学的サンプル（異なる個体、独立培養） | DEG解析のサンプルサイズ設計 |
| **テクニカルリプリケート** | 同一サンプルの複数回測定（ライブラリ重複、レーン重複） | 平均化・フィルタリングの判断 |
| **バッチ効果** | 実験バッチ間の系統的差異（日付、オペレーター、試薬ロット） | PCAで確認、統計モデルに共変量として含める |
| **交絡因子** | 処理群とバッチが一致してしまう設計ミス | 実験開始前に防ぐしかない |
| **擬似リプリケート** | テクニカルリプリケートをバイオロジカルと誤認 | 自由度の過大評価、偽陽性増加 |

**推奨:** §12「データ処理の実践」の冒頭に「解析の前提 — 実験デザインの語彙」として0.5〜1節を追加。または🧬コラムとして§12の統計検定セクションの直前に配置。読者がAIに「このデータセットは3バイオロジカルリプリケート×2バッチだが、バッチ効果を補正した上でDEG解析して」と指示できるレベルを目標とする。

### 中優先度

#### 3. 正規化手法の選択基準

**現状:** §10でCPMとTPMの計算方法をコード例付きで解説しているが、FPKM/RPKMとの違い、手法の選択基準が不足。

**なぜ不足か（3層フィルター）:**
- **判断力**: CPM、TPM、FPKM、DESeq2のmedian-of-ratiosのどれを使うべきかの判断ができない
- **レビュー力**: AIがFPKMで正規化したコードを生成した際、TPMのほうが適切ではないかと疑えない
- **指示力**: 「サンプル間比較にはTPMを使って」「DESeq2の正規化を適用して」と正確に指示できない

**カリキュラムでの扱い:**
- EMBL-EBI: RNA-seq解析コースで正規化（QC, normalization, differential expression analysis）を扱う
- CBW: RNA-seq Analysisワークショップで正規化手法を比較

**追加すべき判断基準:**
| 手法 | 補正対象 | 使うべき場面 | 避けるべき場面 |
|---|---|---|---|
| CPM | ライブラリサイズのみ | 同条件・同遺伝子長の比較 | 遺伝子間比較（長さバイアス） |
| TPM | ライブラリサイズ + 遺伝子長 | サンプル間の遺伝子比率比較 | DEG検定の入力（過剰補正） |
| FPKM/RPKM | ライブラリサイズ + 遺伝子長（旧手法） | レガシーデータとの互換 | 新規解析（TPMに置換推奨） |
| DESeq2 normalization | ライブラリサイズ + 組成バイアス | DEG解析 | 可視化目的（TPMのほうが直感的） |

**推奨:** §10の正規化セクションに上記の判断基準表を追加（0.5節未満の追記）。既存のCPM/TPM解説の直後に配置。

#### 4. ゲノムアノテーションの概念的理解

**現状:** §4の🧬コラムでGFF3/GTFのフィーチャー型（CDS, gene, mRNA, exon, intron）をリスト、§18でEnsembl ID体系（ENSG/ENST/ENSP）とRefSeqプレフィクス（NM_/NR_/XM_）を解説。しかし、遺伝子構造の生物学的意味とID体系間のマッピングが不足。

**なぜ不足か（3層フィルター）:**
- **指示力**: 「gene-levelでカウントして」vs「transcript-levelでカウントして」の指示の違いを理解できない
- **レビュー力**: AIがfeatureCountsでexonレベルでカウントしたコードを生成したとき、geneレベルが適切ではないかと判断できない
- **判断力**: リファレンスアノテーションとしてGENCODEとRefSeqのどちらを使うかを選べない

**カリキュラムでの扱い:**
- MIT 7.91J: RNA-seq解析でisoform-level analysisを扱う
- EMBL-EBI: Ensemblコースでアノテーション体系を詳細に教育
- GTN: Genome Annotationトピックカテゴリが独立して存在

**本書で不足している概念:**
| 概念 | 現状 | 必要な追加 |
|---|---|---|
| Gene vs Transcript | 未説明 | 1つの遺伝子が複数の転写産物を持つ理由（選択的スプライシング） |
| exon/intron/UTR/CDS | §4でリスト | 生物学的意味と相互関係の図 |
| Gene-level vs Transcript-level counting | 未説明 | RNA-seqのカウント粒度の選択基準 |
| GENCODE vs RefSeq | 個別に説明 | 選択基準（ヒトならGENCODE推奨、他種ならRefSeq等） |
| ID変換 | 未説明 | BioMart/gprofiler2/MyGene.infoでのID変換パターン |

**推奨:** §4の🧬コラム「バイオインフォの主要フォーマット」を拡張し、GFF3/GTFの説明の直後に「遺伝子構造の基礎知識 — なぜフォーマットの理解に生物学が必要か」として4〜5段落追加。または§18のアノテーションDB（Ensembl）の節に統合。

#### 5. リファレンスゲノム・アノテーションの選択

**現状:** 本書全体を通じて、リファレンスゲノムのバージョン選択やアノテーションソースの選択に関する議論がない。

**なぜ不足か（3層フィルター）:**
- **判断力**: AIに「ヒトゲノムにマッピングして」と指示するとき、hg38 vs GRCh38 vs T2T-CHM13の違いを知らないと不適切なリファレンスを使うリスク
- **指示力**: 「GENCODE v44のGTFを使って」「RefSeqのGFF3を使って」と具体的に指定できない
- **レビュー力**: AIが古いバージョンのリファレンスを使ったコードを生成しても気づけない

**カリキュラムでの扱い:**
- EMBL-EBI: Ensemblコースでゲノムアセンブリバージョンを扱う
- CBW: 各解析ワークショップでリファレンスの指定が前提

**本書で不足している概念:**
| 概念 | 内容 |
|---|---|
| **ゲノムアセンブリバージョン** | hg19 vs hg38, mm10 vs mm39, GRCh37 vs GRCh38 vs T2T |
| **アノテーションソース** | GENCODE vs RefSeq vs Ensembl, バージョン番号の重要性 |
| **座標のリフトオーバー** | 異なるアセンブリ間の座標変換（liftOver/CrossMap） |
| **リファレンス選択の影響** | マッピング率、バリアントコール結果への影響 |

**推奨:** §4-2「フォーマット選択の判断基準」の末尾に🧬コラム「リファレンスゲノムの選び方」を追加（0.5〜1ページ）。または§18のデータベース章の冒頭で扱う。

#### 6. 前処理の判断基準（トリミング・フィルタリング）

**現状:** §12のRNA-seqパイプラインにトリミングステップ（Trimmomatic）があり、§5のツール一覧にfastpがあるが、トリミングのパラメータ選択基準やフィルタリング判断の指針がない。

**なぜ不足か（3層フィルター）:**
- **判断力**: AIが`LEADING:3 TRAILING:3 SLIDINGWINDOW:4:15 MINLEN:36`を生成したとき、各パラメータの意味と妥当性を評価できない
- **指示力**: 「低品質リードを除去して」ではなくQualityスコアの閾値を指定できない
- **レビュー力**: AIが過剰にトリミングした（リードが短くなりすぎた）際に気づけない

**カリキュラムでの扱い:**
- GTN: Sequence Analysis > Quality Controlチュートリアルで詳細に扱う
- EMBL-EBI: NGS Bioinformaticsコースでread preprocessing

**追加すべき判断基準:**
| 判断ポイント | 選択肢 | 基準 |
|---|---|---|
| QCツール | FastQC / fastp / MultiQC | fastpならQC+トリミング一括。MultiQCは多サンプル統合レポート |
| 品質トリミング閾値 | Q20 / Q30 | Q20: 99%精度。Q30: 99.9%精度。一般にQ20で十分 |
| アダプター除去 | 自動検出 / 手動指定 | fastpは自動検出。Trimmomatic/cutadaptは手動指定 |
| 最小リード長 | 36bp / 50bp | マッピングツールの最小要件に依存 |
| 低発現遺伝子フィルタリング | CPM > 1 in N samples | Nは最小群のサンプル数 |

**推奨:** §12のパイプライン構成の中で、QCとトリミングの段落を2〜3段落拡充。既存のFastQC→Trimmomatic→STARの流れの中に判断基準を組み込む。

### 低優先度

#### 7. アラインメントツールの使い分け

**現状:** §5のツール一覧にminimap2、§12のパイプラインにSTARが登場。§5の🧬コラムにBLAST+、MAFFT、MUSCLE等のツール一覧がある。しかし「いつどのツールを使うか」の選択基準がない。

**なぜ不足か（3層フィルター）:**
- **指示力**: 「ショートリードのDNA-seqにはBWA-MEM2、RNA-seqにはSTAR、ロングリードにはminimap2」と指定できる
- **判断力**: AIがBowtie2でRNA-seqを処理しようとしたとき（スプライスアウェアでないため不適切）に気づける

**判断基準表:**
| データ種別 | 推奨ツール | 理由 |
|---|---|---|
| DNA-seq ショートリード | BWA-MEM2 / Bowtie2 | 連続マッピング前提 |
| RNA-seq ショートリード | STAR / HISAT2 | スプライスジャンクション対応 |
| ロングリード（PacBio/ONT） | minimap2 | 長いリード・高エラー率対応 |
| アセンブリ後の比較 | minimap2 / nucmer | 大規模配列同士のアラインメント |
| タンパク質相同性検索 | BLAST+ / DIAMOND / MMseqs2 | 速度とデータベースサイズで選択 |
| マルチプルアラインメント | MAFFT / MUSCLE / Clustal Omega | 配列数と精度で選択 |

**推奨:** §5の🧬コラム「車輪の再発明を防ぐ」に2〜3行の選択基準を追加。または§12のパイプライン章にマッピングツール選択の段落を追加。工数が小さく、効果が高い。

#### 8. 統計検定の概念的語彙

**現状:** §10で`scipy.stats`を使った統計検定とBH補正が言及されるが、検定の選択基準（パラメトリック vs ノンパラメトリック、対応あり vs なし）が薄い。

**なぜ不足か（3層フィルター）:**
- **判断力**: 2群比較でt検定かMann-Whitney U検定かの選択
- **レビュー力**: AIがt検定を使ったが、データが正規分布に従わない場合に気づける
- **指示力**: 「ノンパラメトリック検定を使って」と指定できる

**カリキュラムでの扱い:**
- Harvard MCB112: 統計検定に3週分を費やす（P値、ベイズ推論、回帰）
- CBW: "Help! What Statistical Model Should I Use?" ワークショップが存在

**推奨:** §10の統計検定セクションに簡易判断フローチャートを追加。「正規分布？→はい:t検定/ANOVA、いいえ:Mann-Whitney/Kruskal-Wallis」「対応あり？→Wilcoxon signed-rank」程度の概要。深入りせず、AIへの指示に必要な語彙の提供に留める。

#### 9. FAIR原則の明示的言及

**現状:** §6でZenodo/DOI/CITATION.cff、§4でGEO/SRAメタデータ設計、§20でコラボレーション、付録Dで投稿前チェックリストと、FAIR原則の実践はよくカバーされている。しかし「FAIR原則」という用語自体が登場しない。

**なぜ不足か（3層フィルター）:**
- **指示力**: 「FAIRの原則に従ったデータ管理を設計して」とAIに指示できると、メタデータ設計の品質が上がる
- **判断力**: FAIRの4要素（Findable, Accessible, Interoperable, Reusable）を知っていれば、データ管理の設計判断に使える

**カリキュラムでの扱い:**
- GTN: "FAIR Data, Workflows & Research" が独立トピックとして存在
- CBW: "Reproducible Research: Essentials for Managing Your Data" でFAIRを扱う
